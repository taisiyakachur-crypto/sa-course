📝 Модуль BA-03

Вимоги та User Stories:
Декомпозиція, пріоритизація та архітектура вимог

Вимоги — головний продукт BA. Розуміння різних типів вимог і методів їх збору є ключовою компетенцією.

📝 Типи вимог
ТипЩо описуєПрикладХто використовує
Business RequirementsЧому ініційована зміна, стратегічні цілі"Збільшити конверсію реєстрацій на 20%"Топ-менеджмент, PO
Stakeholder RequirementsЯкі результати потрібні конкретним групам"Менеджер бачить свої замовлення"BA, PO
Solution RequirementsЩо система повинна робити (функціональні)"Система надсилає email при реєстрації"Dev, QA
Non-Functional RequirementsЯк добре система виконує функції"Відповідь API ≤ 200ms при 1000 RPS"Dev, Architect
Transition RequirementsТимчасові вимоги для переходу"Міграція даних зі старої системи"BA, DBA, Dev
🎯 User Story — формат та якість
Формат: As a [роль], I want [дія], so that [цінність / мета]
✅ Добра User Story

"As a registered user, I want to reset my password via email, so that I can regain access if I forget it."

  • Чітка роль (registered user)
  • Конкретна дія (reset via email)
  • Зрозуміла цінність (regain access)
❌ Погана User Story

"As a user, I want the system to work better."

  • Роль розмита — просто "user"
  • Дія незрозуміла — "work better"?
  • Цінність відсутня
✅ INVEST — перевірка якості User Story
I

Independent

Незалежна від інших stories. Можна розробляти у будь-якому порядку.

N

Negotiable

Деталі обговорюються — не жорсткий контракт а відправна точка.

V

Valuable

Несе реальну цінність для користувача або бізнесу.

E

Estimable

Команда може оцінити складність. Якщо ні — потрібно деталізувати.

S

Small

Завершується за один спринт. Великі stories потрібно ділити.

T

Testable

Можна написати тест. Якщо ні — вимоги недостатньо специфічні.

🎯 Acceptance Criteria — Given / When / Then

📋 Приклад: Password Reset

## Acceptance Criteria Scenario 1: Успішне скидання пароля Given користувач на сторінці Login When він клікає "Forgot Password" і вводить валідний email Then система надсилає лист протягом 2 хвилин And посилання активне 1 годину And показує: "Check your email" Scenario 2: Email не зареєстрований Given користувач вводить незареєстрований email When натискає Submit Then система показує: "Check your email" -- ВАЖЛИВО: не розкриваємо існування акаунту! Scenario 3: Прострочене посилання Given посилання старше 1 год Then показує помилку "Link expired" з кнопкою "Try again"
