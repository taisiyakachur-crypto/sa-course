🔌 Модуль API-02

API — проектування та нотації:
REST, GraphQL, gRPC, WebSocket та Webhooks

API — це контракт між системами. BA що розуміє REST і GraphQL пише точніші вимоги до інтеграцій і може самостійно тестувати через Postman.

🌐 Що таке API

API (Application Programming Interface) — це "меню ресторану". Ти бачиш що можна замовити (endpoints), але не бачиш як готується страва (server code). API визначає як системи спілкуються між собою.

✅ HTTP Status Codes

2xx — Успіх

200 OK — Запит успішний
201 Created — Ресурс створено
204 No Content — Видалено успішно

4xx — Помилка клієнта

400 Bad Request — Невалідний запит
401 Unauthorized — Не авторизований
403 Forbidden — Немає доступу
404 Not Found — Не знайдено
422 Unprocessable — Помилка валідації

5xx — Помилка сервера

500 Internal Error — Сервер "впав"
502 Bad Gateway — Проблема proxy
503 Unavailable — Сервер недоступний
🎯 BA в Acceptance Criteria: "При успішній реєстрації система повертає 201. При дублікаті email — 409 Conflict. При невалідних даних — 422 з описом полів що не пройшли валідацію."
🔬 REST vs GraphQL vs gRPC vs WebSocket
Тип APIКоли використовуватиПеревагиНедоліки
REST90% веб-додатків, публічні APIПростий, стандартний, легко тестувати в PostmanOver/under fetching
GraphQLСкладні дані, мобільні з обмеженим трафікомГнучкість — запитуєш тільки потрібні поляСкладніше кешування
gRPCВнутрішні мікросервіси, high-performanceШвидкий, бінарний протокол, строга типізаціяНе підходить для браузера
WebSocketЧати, live-дані, ігри, біржіДвостороннє постійне з'єднання, real-timeСкладніша інфраструктура
WebhooksСповіщення про події (платежі, зміни статусу)Push-модель — сервер сам сповіщає клієнтаПотрібен публічний URL
📄 JSON — формат даних API

Запит (POST /api/users):

{ "name": "Олена Петренко", "email": "olena@company.ua", "role": "business_analyst", "active": true }

Відповідь (201 Created):

{ "id": 4521, "name": "Олена Петренко", "email": "olena@company.ua", "role": "business_analyst", "createdAt": "2026-03-19T10:30:00Z" }
🔧 Design-First підхід — OpenAPI / Swagger

Design-first: спочатку описуємо API контракт у Swagger, погоджуємо з командою — потім розробляємо. Мінімізує помилки та переробки.

# Приклад OpenAPI 3.0 специфікації openapi: 3.0.3 info: title: Orders API version: 1.0.0 paths: /orders: get: summary: Отримати список замовлень parameters: - name: status in: query schema: {type: string, enum: [new, paid, delivered]} - name: limit in: query schema: {type: integer, default: 20} responses: '200': {description: Список замовлень} '401': {description: Не авторизований}
🎯 BA роль: BA бере участь у розробці OpenAPI специфікації — описує бізнес-логіку endpoint-ів, обов'язкові поля, можливі помилки та acceptance criteria прямо в специфікації.
❓ Питання для самоперевірки
  • Яка різниця між GET і POST? Коли використовувати PATCH замість PUT?
  • Що означає статус 401 і чим він відрізняється від 403?
  • Коли обрати GraphQL замість REST? Наведи конкретний сценарій.
  • Що таке WebSocket і коли він потрібен замість звичайного API?
  • Як BA включає HTTP статус-коди в Acceptance Criteria?
