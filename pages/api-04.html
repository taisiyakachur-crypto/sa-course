SOAP та SRS документ:
WSDL, XML Schema та стандарти IEEE 830

SOAP зустрічається в legacy та enterprise системах. SRS — головний документ який BA виробляє для технічної команди.

🧼 SOAP Web Services

SOAP (Simple Object Access Protocol) — XML-протокол для передачі структурованих повідомлень. Використовується в банках, держ. системах, ERP де важлива надійність і безпека.

✅ Переваги SOAP

  • Стандартизована безпека (WS-Security)
  • Підтримка транзакцій та reliability
  • Незалежність від транспорту (HTTP, SMTP)
  • Строгий контракт (WSDL)

❌ Недоліки SOAP

  • Велика XML-розмірність повідомлень
  • Складність реалізації
  • Повільніший ніж REST/gRPC
  • Важко тестувати без спеціальних інструментів
<!-- Приклад SOAP запиту --> <soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/"> <soap:Header/> <soap:Body> <m:GetOrderStatus xmlns:m="http://example.org/orders"> <m:OrderID>12345</m:OrderID> </m:GetOrderStatus> </soap:Body> </soap:Envelope>
📄 SRS — Software Requirements Specification

SRS — це документ що є угодою між замовником і командою: що система має робити і як. Стандарт IEEE 830 / ISO 29148 визначає структуру та якісні характеристики.

🎯

Точність

Кожна вимога описує одну конкретну річ без двозначності. "Система має" замість "система може".

✅

Перевіряємість

Кожну вимогу можна протестувати. Якщо не можна — вона сформульована неправильно.

🔗

Трасованість

Кожна вимога пов'язана з бізнес-ціллю і тест-кейсом. Зміна вимоги → оновлення тестів.

📊

Пріоритетність

Must/Should/Could (MoSCoW). Команда знає що є критичним а що — бажаним.

🏗 Структура SRS документа
1

Вступ

Мета документа, scope, терміни та скорочення, аудиторія, посилання на суміжні документи.

2

Загальний опис

Контекст системи, ролі користувачів, припущення і обмеження. Де система буде використовуватись.

3

Функціональні вимоги (FR)

Що система повинна робити. User Stories з Acceptance Criteria. CRUD операції, бізнес-логіка, flows.

4

Нефункціональні вимоги (NFR)

Як добре: Performance ≤ 200ms, Availability 99.9%, Security TLS, Usability WCAG 2.1.

5

Зовнішні інтерфейси

API специфікація (OpenAPI), UI wireframes, інтеграції з зовнішніми системами.

6

Обмеження та припущення

Технічний стек, залежності, що НЕ входить до scope (out of scope).

7

Додатки

Глосарій, діаграми (ER, BPMN, sequence), матриця трасованості вимог.

📋 Шаблон фрагменту SRS
# SRS — Модуль авторизації ## FR-1: Авторизація через email + пароль As a registered user I want to log in with email and password So that I can access my personal dashboard Acceptance Criteria: Given user enters valid credentials When clicks "Login" Then system returns JWT token (200 OK) And redirects to /dashboard Given user enters wrong password When clicks "Login" 5th time Then account blocked for 15 min (429) ## NFR-1: Performance - Login response time ≤ 500ms at 1000 concurrent users - System availability ≥ 99.9% (SLA) ## NFR-2: Security - Passwords hashed with bcrypt (cost ≥ 12) - 2FA required for admin role - All traffic via HTTPS/TLS 1.2+

Глосарій
SOAPSimple Object Access Protocol — XML-протокол для обміну структурованими повідомленнями
WSDLWeb Services Description Language — XML-опис контракту SOAP сервісу
SRSSoftware Requirements Specification — документ з повними вимогами до системи
NFRNon-Functional Requirements — вимоги до якості: продуктивність, безпека, масштабованість
Traceability MatrixМатриця зв'язку між вимогами, User Stories та тест-кейсами
ScopeМежі проєкту — що входить і що НЕ входить у розробку
