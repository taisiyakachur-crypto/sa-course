Безпека API та типи даних:
OAuth2, JWT, RBAC, Rate Limiting та помилки

Безпека — не опція. BA включає security requirements у кожну специфікацію і розуміє типи авторизації та форматування помилок.

🔑 Аутентифікація vs Авторизація

🪪 Аутентифікація

ХТО ТИ? Перевірка особи користувача.

  • Логін + пароль (bcrypt хешування)
  • 2FA / MFA (SMS, TOTP)
  • OAuth2 (Login with Google/GitHub)
  • SSO — Single Sign-On
BA вимога: "2FA обов'язково для admin ролей"

🛡 Авторизація (RBAC)

ЩО ТИ МОЖЕШ? Перевірка дозволів.

  • Role-Based Access Control
  • Матриця ролей і дозволів
  • Перевірка на кожному endpoint
BA вимога: "Менеджер бачить тільки свої проєкти"
🔑 JWT — JSON Web Token

JWT — електронний жетон що містить дані користувача і підпис. Stateless авторизація — не потрібно зберігати сесії на сервері.

// JWT структура: header.payload.signature // Payload містить claims: { "sub": "user_123", // User ID "role": "business_analyst", // Роль "email": "olena@co.ua", // Email "exp": 1711843200 // Час закінчення (15 хв) } // Запит з токеном: GET /api/orders Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
🎯 BA вимоги: "Access token дійсний 15 хвилин. Refresh token — 30 днів. При logout — токен інвалідується. Зберігати тільки в httpOnly cookie."

🚫 Rate Limiting та захист від атак
⏱

Rate Limiting

Обмеження кількості запитів: 100 req/хв на IP. Захист від DDoS та брутфорсу.

BA вимога: "Максимум 5 спроб входу, потім блокування на 15 хв"
🔒

HTTPS / TLS

Шифрування всього трафіку між клієнтом і сервером. TLS 1.2+ обов'язково.

BA вимога: "Весь трафік тільки через HTTPS, HTTP редиректиться"
💉

SQL Injection / XSS

Атаки через поля вводу. Захист: prepared statements, input validation, CSP заголовки.

BA вимога: "Валідація всіх user-input полів на backend"
🛡

CORS

Cross-Origin Resource Sharing — контроль які домени можуть робити запити до API.

BA вимога: "API доступний тільки з our-domain.com"
⚠️ Формат помилок API

Структуровані помилки допомагають frontend показувати правильні повідомлення користувачам і допомагають розробникам дебажити.

Погана помилка:

// Нічого не зрозуміло { "error": "Error 422" }

Хороша помилка (RFC 7807):

{ "error": { "code": "VALIDATION_ERROR", "message": "Email невалідний", "field": "email", "details": [{ "field": "email", "message": "Invalid format" }] } }
